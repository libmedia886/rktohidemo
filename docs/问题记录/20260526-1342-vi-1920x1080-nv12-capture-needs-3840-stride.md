# VI 1920x1080 NV12 capture needs 3840 stride

metadata:
- date: 2026-05-26 13:42:12
- kind: runtime
- status: fixed
- repository: /userdata/alldemo
- summary: The camera VI path failed QBUF at 1920x1080 when the standalone page used stride 1920; stride 3840 works on this device.

## Symptom

VI: QBUF 失败: Invalid argument

## Reproduction

```bash
timeout -s INT -k 3s 8s ./scripts/run_alldemo.sh --only CAP_DEHAZE
```

## Evidence

- MEDIA_VI_SetAttr: ok dev=0 device=/dev/video-camera0 1920x1080 fps=30 buf=4 pool=2 format=1
- Final CAP validation used Source: output (1920x1080, stride=3840, type=1).

## Affected Files

- src/pages/page_cap_dehaze_offline.c

## Root Cause

The camera driver requires the 1920x1080 NV12 buffer stride used by this media stack to be 3840 bytes, so a 1920-byte Y stride under-allocates or mismatches QBUF expectations.

## Fix Or Workaround

Use CAP_LIVE_SRC_STRIDE 3840 and size the VI pool from stride * height * 3 / 2.

## Recurrence Prevention

For live VI pages, copy the validated camera stride from the original demo path and verify QBUF, not only MEDIA_VI_SetAttr.
